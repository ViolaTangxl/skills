## 章节设计

### 第一章：开篇 — 专业能力的门槛问题 (约 500 字)

**目的**：从范式转变的角度引出问题

**内容要点**：
1. **Hook**：技术领域存在大量"专家专属"的能力
   - 混沌工程/Resilience 测试就是典型例子
   - 大家都知道重要（Netflix Chaos Monkey 的故事），但实际做的团队很少
   - 为什么？不是不想做，而是**门槛太高**

2. **问题本质**：专业工具需要专业知识
   - AWS FIS 是很强大的混沌工程服务
   - 但要用好它，需要先成为"FIS 专家"
   - 学习成本成为采用的最大障碍
   - 这是很多专业工具的共同困境

3. **范式转变的契机**：AI Agent 带来的可能性
   - 如果专业知识可以被"封装"到 Agent 中呢？
   - 用户只需描述意图，Agent 负责选择工具、配置参数、执行操作
   - 这不是"替代专家"，而是"让非专家也能做专家的事"

4. **预告**：本文以混沌工程为例，展示这种范式转变的具体实践
   - 三个 Claude Code Agent Skill 如何让任何工程师都能完成 EKS 故障演练
   - 从"先学 FIS 再做实验"到"描述意图就能做实验"

### 第二章：旧范式的困境 — 为什么混沌工程难以普及 (约 300 字)

**目的**：具体说明专业工具的门槛问题，为新范式做铺垫

#### 痛点 1：FIS 学习门槛高

**内容要点**：
- AWS FIS 提供了丰富的故障注入能力，但学习曲线陡峭：
  - **Action 众多**：几十种 action，每种对应不同服务、不同故障类型
    - 例：`aws:rds:failover-db-cluster` vs `aws:rds:reboot-db-instances` 有什么区别？
    - 例：想测试 EKS Pod 网络延迟，应该用哪个 action？需要什么前置条件？
  - **Scenario Library 复杂**：4 种复合场景（AZ Power Interruption、AZ Application Slowdown 等），每种包含多个子 action，配置结构复杂
  - **资源兼容性隐蔽**：某些 action 只适用于特定资源类型（如 failover 只能用于 Aurora 集群，不能用于普通 RDS 实例），配置错误只有在实验启动时才会发现
- **新手的困境**：从"我想测 AZ 故障"到写出正确的 experiment template，中间需要：
  1. 理解 FIS 概念模型
  2. 找到合适的 action 或 scenario
  3. 了解目标资源的配置要求
  4. 编写正确的 JSON 模板
  - 这条路对不熟悉 FIS 的人来说很长

#### 痛点 2：应用层影响的观测盲区

**内容要点**：
- FIS 报告只有基础设施视角
  - 例：RDS failover 耗时 30 秒 ✓
- 应用层问题无法回答：
  - 连接池断开多久？
  - 重试成功了吗？
  - 用户看到错误了吗？
- 手工收集 kubectl logs + 时间线对齐 = 繁琐且容易遗漏

### 第三章：新范式的实践 — AI Agent 如何封装专业能力 (约 1200 字)

**目的**：展示 Agent Skill 如何将专业知识封装成可对话的能力

#### 整体架构

**内容要点**：
- 三个 Claude Code Agent Skill 形成流水线：
  ```
  prepare → execute → log-analysis
  ```
- 用户通过自然语言与 Agent 交互
- Agent 调用 AWS CLI 执行操作

**配图**：端到端工作流架构图（三个 skill + 输入输出 + 生成的文件）

#### Skill 1：aws-fis-experiment-prepare

**解决痛点**：#1 FIS 学习门槛高

**内容要点**：
- 用户用自然语言描述意图，无需了解 FIS 细节
  - 示例："准备一个 AZ 断电实验，目标是 us-east-1a"
  - 示例："我想测试 RDS 故障转移对应用的影响"
- Agent 自动执行：
  1. 理解用户意图，选择合适的 action 或 scenario
  2. 查询 AWS 资源，识别目标并验证兼容性
  3. 读取 FIS Scenario Library 文档（复合场景的模板无法通过 API 生成）
  4. 生成完整的配置文件（experiment template、IAM policy、CloudFormation 等）
  5. 部署 CloudFormation
- **核心价值**：用户不需要学习 FIS action 细节，只需描述"想测什么"
- **额外亮点**：自愈式 CFN 部署 — 部署失败时自动分析错误、修复模板、删除失败栈、重试（最多 5 次），减少调试时间

#### Skill 2：aws-fis-experiment-execute

**功能**：安全执行 + 监控

**内容要点**：
- 验证 CFN 栈状态，提取实验模板 ID
- **安全机制**：强制用户确认才启动
  - 展示影响范围、目标资源、预计时长
- 轮询实验状态，提醒查看 Dashboard
- 生成结果报告：
  - 按服务分组的影响分析
  - 每个服务的时间线、观察、关键发现

#### Skill 3：eks-app-log-analysis

**解决痛点**：#2 应用层观测盲区

**内容要点**：
- 两种模式：
  - 实时监控：实验进行中，每 30 秒展示日志洞察
  - 事后分析：实验结束后批量分析
- 自动收集相关应用的 kubectl logs
- 生成应用层分析报告：
  - 错误时间线
  - 错误模式统计
  - 恢复时间
  - 与 FIS 事件的关联
- **关键亮点**：补全了基础设施报告缺失的"应用视角"

### 第四章：范式转变的效果 (约 400 字)

**目的**：展示门槛降低带来的实际变化

#### 对比：谁能做混沌工程？

| 维度 | 旧范式（手工） | 新范式（Agent Skill） |
|------|---------------|----------------------|
| **前置知识** | 需要熟悉 FIS action、scenario、资源兼容性 | 只需描述测试意图 |
| **学习曲线** | 数小时到数天 | 几乎为零 |
| **执行时间** | 半天到一天（新手） | 10-15 分钟 |
| **谁能做** | FIS 专家或愿意投入学习的工程师 | 任何了解业务的工程师 |

#### 关键变化
- 不是"快了 10 倍"，而是"原本不会做的人现在能做了"
- 混沌工程从"专家任务"变成"团队常规实践"的可能性

### 第六章：更大的图景 — 专业能力的普惠化 (约 400 字)

**目的**：升华到技术范式转变的更广视角

**内容要点**：

1. **混沌工程只是一个例子**
   - 技术领域有大量"专家专属"的能力：安全审计、性能调优、合规检查、架构评估...
   - 这些能力的共同特点：工具强大，但学习门槛高
   - AI Agent 提供了一种新的可能：把专业知识封装成可对话的能力

2. **范式转变的本质**
   - 旧范式：人先学工具 → 再用工具做事
   - 新范式：人描述意图 → Agent 选择工具、配置参数、执行操作
   - 这不是"降低标准"，而是"改变交互方式"
   - 专业知识没有消失，而是被封装到了 Agent 中

3. **对工程师的意义**
   - 不是被替代，而是被赋能
   - 可以专注于**决策和分析**（这是混沌工程的真正价值），而不是**工具配置**
   - 团队可以更频繁地进行 Resilience 测试，而不是因为门槛高而放弃

4. **展望**
   - Agent Skill 是一种新的知识封装和分发方式
   - 期待更多专业领域的能力被"普惠化"

## 写作风格指南

1. **语气**：专业但不晦涩，技术决策者和实践者都能读懂
2. **避免**：过度营销语言、空洞的 buzzword
3. **代码示例**：只在必要时展示关键命令或配置片段，不要大段代码
4. **中文习惯**：
   - 专有名词保持英文（AWS FIS, CloudFormation, EKS）
   - 中英文之间加空格
   - 避免翻译腔
